Instruction réseau manquante

Dans la Figure 1, l'interface GigabitEthernet 0/1 de R1 vient d'être configurée avec l'adresse 10.10.10.1/24 et est désormais active. Une contiguïté de voisinage EIGRP a été établie entre R1 et R3 qui échangent des informations de routage. Cependant, comme le montre la Figure 2, un test ping envoyé depuis le routeur R3 vers une interface G0/1 du réseau 10.10.10.1 de R1 a échoué.

Dans la Figure 3, le résultat de la commande show ip protocols sur le routeur R1 montre que le réseau 10.10.10.0/24 n'a pas été annoncé aux voisins EIGRP. Par conséquent, le réseau 10.10.10.0 doit être annoncé.

Dans la Figure 4, le processus EIGRP de R1 est configuré pour annoncer le réseau 10.0.0.0 par classe.

La Figure 5 confirme l'ajout de la route 10.10.10.0/24 à la table de routage de R3. L'accessibilité est vérifiée et confirmée à l'aide d'une requête ping ciblant l'interface GigabitEthernet 0/1 de R1.

Protocole EIGRP pour IPv6

Des commandes et des critères de dépannage analogues s'appliquent au protocole EIGRP pour IPv6.

Voici les commandes équivalentes à utiliser avec le protocole EIGRP pour IPv6 :

  • Router# show ipv6 protocols

  • Router# show ipv6 route

Pour ajouter une instruction réseau manquante sous IPv6, utilisez la commande ipv6 eigrp autonomous-system en mode de configuration d'interface :

Router(config-if)# ipv6 eigrp autonomous-system

Remarque : une autre forme de route manquante peut entraîner le filtrage des mises à jour de routage entrantes et sortantes. Les listes de contrôle d'accès assurent le filtrage pour plusieurs protocoles et peuvent avoir un impact sur l'échange de messages de protocole de routage. Cela peut expliquer l'absence de certaines routes dans la table de routage. Utilisez la commande show ip protocols pour afficher les éventuelles listes de contrôle d'accès appliquées à EIGRP. Le filtrage des mises à jour de routage n'est pas abordé dans ce programme.
